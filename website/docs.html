System Protocols

Cognito is an objective workspace for local machine intelligence. This guide provides an exhaustive technical standard for the system's architecture and operation.

01 Architecture

The system is built on a decoupled architecture composed of three primary layers:

  • Application Layer: Electron-based container managing window lifecycle and IPC bridge.
  • Intelligence Layer: FastAPI backend localized to the host machine, wrapping llama-cpp-python.
  • Retrieval Layer: In-memory TF-IDF vectorization engine for document context and agentic web-search handlers.

02 Setup

Cognito requires local runtime preparation. We favor source-level deployment to ensure system integrity and auditability.

Prerequisites

  • Node.js (v18.0.0+): Required for the React interface.
  • Python (v3.10.0+): Required for neural weights processing and API services.
  • Build Tools: C++ compiler (Xcode on Mac, VS Build Tools on Windows) for compiling llama-cpp bindings.

Initial Boot Sequence

# Protocol 01: Remote Data Acquisition
git clone https://github.com/ArjunDeshwal/cognitoai.git
cd cognitoai

# Protocol 02: Intelligence Environment
python -m venv venv
# Unix: source venv/bin/activate | Win: venv\Scripts\activate
pip install -r backend/requirements.txt

# Protocol 03: Interface Deployment
cd app
npm install
npm run electron:dev

03 Document RAG

Cognito implements a Retrieval-Augmented Generation (RAG) system for local knowledge injection.

Mechanism

Uploaded documents (PDF, TXT) are processed using PyMuPDF. The system utilizes a TF-IDF Vectorizer to generate sparse embeddings for text chunks. Upon query, a Cosine Similarity analysis is performed to retrieve the Top-K relevant segments for the neural context window.

Constraints

  • Chunking: Default 400 words with a 50-word overlap.
  • Locality: Vector indices are stored in-memory and are not persisted to disk, ensuring zero-trace document analysis.

04 Agentic Search

The system features an autonomous search capability where the model itself decides when to access external information.

Operational Flow

  1. The model generates a [SEARCH: query] token if it identifies a factual gap.
  2. The backend triggers the web_search utility.
  3. Results are injected into the system prompt as [SEARCH RESULTS].
  4. The model completes its reasoning based on the retrieved context.
DEEP SEARCH: Enabling Deep Search mode triggers a query decomposition algorithm that spawns multiple sub-queries to map complex information landscapes.

05 Model Management

Cognito supports the GGUF (GPT-Generated Unified Format) standard.

  • Quantization: Recommended Q4_K_M or Q5_K_M for optimal performance-to-vRAM ratio.
  • Acceleration: Backends are pre-configured for Metal (Apple Silicon) and CUDA (NVIDIA). Hardware layers are automatically mapped during model load.

06 API Reference

The internal intelligence service exposes the following endpoints for developers:

Model Control

POST/v1/load_model
Loads a GGUF file into the neural execution engine.

Inference

POST/v1/chat/completions
Standard OpenAI-compatible chat endpoint supporting streaming and agentic flags.

Knowledge System

GET/v1/documents — List active docs.
POST/v1/documents/upload — Process new data source.
